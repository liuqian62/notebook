# 初始化与标定

这页想讲清楚一个经常被低估的问题：

很多 SLAM 系统不是“算法不行”，而是从一开始就没有被正确初始化，或者标定和时间同步存在误差。

从工程角度看，下面这些问题经常比前端或后端公式本身更先决定系统能不能跑稳：

- 单目怎么初始化
- VIO 怎么初始化尺度、重力、速度和偏置
- LIO 为什么一定要重视去畸变
- 相机、IMU、LiDAR 的内外参怎么来
- 时间同步误差为什么会直接拖垮结果

## 1. 为什么初始化和标定这么重要

SLAM 不是一帧一帧独立求解，它是一个强依赖历史状态的递推系统。

这意味着：

- 初始姿态错了，后面都会带偏
- 外参错了，传感器之间的几何关系一直在“内耗”
- 时间不同步，传感器观测实际上对应的不是同一时刻

所以很多“看起来像优化问题”的坏结果，根因其实是：

- 初始化失败
- 标定不准
- 时间戳不对

## 2. 先区分三类“初始化”

### 2.1 几何初始化

典型于视觉 SLAM，关注的是：

- 初始相机相对位姿
- 初始地图点
- 初始尺度或相对尺度

### 2.2 惯性初始化

典型于 VIO / LIO，关注的是：

- 重力方向
- 速度
- IMU 零偏
- 尺度
- 传感器外参

### 2.3 系统级初始化

这是更工程化的一层，关注：

- 传感器是否都在正常发数据
- 时间戳是否可信
- 外参是否加载正确
- 是否已经具备足够激励

实际项目里，很多系统“起不来”，不是数学初始化没公式，而是系统级初始化没准备好。

## 3. 单目视觉为什么必须初始化

单目视觉一开始只有 2D 图像观测，所以天然存在几个问题：

- 没有直接深度
- 尺度不确定
- 纯旋转时无法稳定恢复三维结构

所以单目系统通常不能像双目 / RGB-D 一样一上来就直接跑 PnP，而是要先完成：

- 两视图几何恢复
- 初始三角化
- 初始地图构建

你的旧笔记里其实已经提到这一点：单目 VO 必须先初始化，然后才能稳定使用 PnP。

## 4. 单目初始化一般怎么做

### 4.1 两视图几何

最常见的是从前两帧或早期关键帧中估计：

- `H` 单应矩阵
- `F` 基础矩阵
- 或进一步恢复 `E` 本质矩阵

然后根据场景情况判断：

- 更像平面 / 纯旋转，还是一般三维场景

### 4.2 恢复相对位姿和初始地图

初始化阶段会尝试恢复：

- 相机相对旋转
- 相机相对平移方向
- 初始稀疏地图点

但要注意：

- 平移只有方向，尺度还不真实

### 4.3 为什么纯旋转会出问题

因为纯旋转下：

- 没有有效视差
- 点的深度不容易被稳定恢复

这也是单目初始化时最经典的退化情形之一。

## 5. VIO 初始化为什么更难

相比单目视觉，VIO 虽然多了 IMU，但初始化反而更复杂，因为它不仅要估计几何，还要同时恢复：

- 尺度
- 重力方向
- 速度
- 陀螺仪 / 加速度计偏置
- 有时还包括相机-IMU 外参和时间偏移

`VINS-Mono` 官方论文摘要明确提到 monocular VINS 面临 IMU processing、estimator initialization、extrinsic calibration 和 nonlinear optimization 等挑战  
来源：<https://researchportal.hkust.edu.hk/en/publications/vins-mono-a-robust-and-versatile-monocular-visual-inertial-state--2>

## 6. VIO 初始化一般在估什么

### 6.1 重力方向

系统需要知道：

- 世界坐标系里“向下”在哪

否则姿态和加速度解释会一直混杂。

### 6.2 速度

仅靠图像，很难直接给出每个时刻真实速度。

但 IMU 状态传播需要速度项，所以初始化要恢复早期滑窗中的速度。

### 6.3 尺度

单目视觉只能恢复 up-to-scale 的结构和轨迹。

加入 IMU 后，若运动激励足够，可以通过视觉和惯性约束共同恢复 metric scale。

### 6.4 IMU 偏置

陀螺和加速度计偏置如果不尽早估出来，会直接污染：

- 预积分
- 姿态传播
- 速度和位置估计

## 7. OpenVINS 和 VINS-Mono 对初始化的提示

`OpenVINS` 文档里专门有 `ov_init`，并明确说明系统支持 static 和 dynamic state initialization  
来源：<https://docs.openvins.com/>

`OpenVINS` 的 `InertialInitializer` 文档还直接写了它会先尝试 dynamic initialization，如果失败再尝试 static initialization；而 dynamic initialization 的核心会先恢复 camera-to-IMU rotation，再恢复 velocity、gravity 和 feature positions  
来源：<https://docs.openvins.com/classov__init_1_1InertialInitializer.html>

这件事很值得记住：

- VIO 初始化不是“等几帧就行”
- 它其实是在解一个早期小规模联合估计问题

## 8. 什么叫“足够激励”

很多初始化失败，根本原因不是代码错，而是运动不够“有信息量”。

### 8.1 对 VIO 来说

如果一开始只是：

- 缓慢平移
- 几乎不转
- 或者长时间静止

那就很难把这些量分开：

- 重力
- 偏置
- 真实加速度
- 尺度

通常更好的初始化动作是：

- 有一定角速度变化
- 有多方向平移
- 不要只有单轴匀速运动

### 8.2 对标定来说

同样也需要激励。

比如相机-IMU 标定时，Kalibr 文档明确建议：

- 激励所有 IMU 轴
- 保持低模糊
- 使用可信时间戳
- 避免只做很单一的运动

来源：<https://github.com/ethz-asl/kalibr/wiki/camera-imu-calibration>

## 9. LIO 里“初始化”为什么经常和去畸变绑在一起

LIO 和 VIO 不完全一样。

它的关键痛点不是单目尺度，而是：

- 一帧 LiDAR 扫描是分时采集的
- 平台在扫描过程中通常正在运动

所以如果不处理 motion distortion，点云本身就已经“弯了”。

这也是为什么很多 LIO 系统里，初始化和 IMU 融合会和下面这些事绑得很紧：

- deskew
- 外参
- 时间同步
- IMU 偏置和重力

## 10. FAST-LIO 对同步和外参的要求

`FAST-LIO` 官方 README 明确提醒：

- IMU 和 LiDAR 必须同步，这很重要
- 点云里每个点的时间戳非常重要，因为前向和后向传播都要用到
- 如果外参已知，推荐关闭在线外参估计
- 时间同步若能硬件做就尽量不要只依赖软件补偿

来源：<https://github.com/hku-mars/FAST_LIO>

这基本说明了 LIO 工程里最重要的几件事：

- 点时间戳
- 外参
- 时间同步

## 11. LIO-SAM / FAST-LIO 里的 LiDAR-IMU 初始化

这类系统一开始通常至少要有：

- LiDAR 到 IMU 的外参
- 合理的 IMU 噪声参数
- 足够稳定的时间戳

如果这些不准，就会连锁影响：

- 去畸变
- scan-to-map 配准
- 姿态预测
- 融合一致性

而 `LI-Init` 这类工作则专门针对 LiDAR-IMU 初始化，目标是估计：

- temporal offset
- extrinsic parameter
- gravity vector
- IMU bias

来源：<https://github.com/hku-mars/LiDAR_IMU_Init>

## 12. 标定通常分哪几层

### 12.1 相机内参

包括：

- 焦距
- 主点
- 畸变参数

如果内参不准，最直接的影响是：

- 重投影误差长期偏大
- 图像边缘区域误差更明显
- 三角化和 PnP 稳定性变差

### 12.2 传感器外参

常见是：

- 相机到 IMU
- LiDAR 到 IMU
- LiDAR 到相机

外参错了，本质上就是把“来自不同坐标系的观测”强行当成一致的。

### 12.3 时间偏移

这是很多人最容易忽略的。

即使空间外参完全正确，如果：

- 图像时间晚了几毫秒
- IMU 时间早了几毫秒

高速运动下也会立刻出明显误差。

## 13. Kalibr 在这里为什么几乎是绕不过去的工具

`Kalibr` 官方 wiki 说明它不仅能做 multi-camera calibration，还能做 camera-IMU 的 spatial and temporal calibration，并支持 IMU intrinsic 参数估计  
来源：<https://github.com/ethz-asl/kalibr/wiki>

`OpenVINS` 的官方 calibration guide 也明确推荐用 Kalibr 做：

- camera intrinsics
- camera-IMU extrinsics
- temporal offset calibration

来源：<https://docs.openvins.com/gs-calibration.html>

所以对于 VIO 来说，一个非常实用的默认选择是：

- 相机内参：Kalibr 或 OpenCV
- 相机-IMU 外参和时间偏移：Kalibr
- IMU 噪声模型：Allan variance 工具 + Kalibr 配置

## 14. 时间同步为什么会直接拖垮系统

想象一下：

- 图像对应的是 `t`
- IMU 实际对应的是 `t + dt`

那在快速运动时，系统会误以为：

- 同一个姿态下看到的图像和 IMU 不一致

结果会表现为：

- 初始化不稳定
- 外参“看起来总在漂”
- 重投影误差大
- 高动态场景下效果明显变差

尤其对下面这些系统最敏感：

- 高速 VIO
- 高频 LIO
- rolling shutter 相机系统

## 15. 在线标定和离线标定怎么取舍

### 15.1 离线标定

优点：

- 结果通常更稳定
- 便于重复实验
- 适合作为默认配置

缺点：

- 传感器装配变化后要重做
- 对设备一致性有依赖

### 15.2 在线标定

优点：

- 能适应小幅变化
- 某些系统启动更灵活

缺点：

- 参数耦合更强
- 可观性更敏感
- 容易在初始化阶段不稳

工程上比较常见的策略是：

- 离线给一个尽量准的初值
- 在线只允许小范围 refinement

## 16. 工程里最常见的坏症状和排查顺序

### 16.1 症状：系统刚启动就发散

优先看：

- 时间戳是否单调正确
- 外参是否坐标系写反
- IMU 单位是否正确
- 噪声参数是否离谱

### 16.2 症状：低速还行，高速就崩

优先看：

- 时间同步
- 运动模糊
- rolling shutter
- LiDAR 点时间戳
- 去畸变质量

### 16.3 症状：轨迹方向对，但尺度或姿态慢慢歪

优先看：

- VIO 初始化质量
- IMU 偏置估计
- 重力方向初始化
- 外参是否有小误差

### 16.4 症状：回环前还行，一融合别的传感器就坏

优先看：

- 外参是否统一
- 多传感器时间基准是否一致
- 坐标系定义是否混乱

## 17. 面试里常被问的几个点

### 17.1 单目为什么必须初始化

标准答法：

- 因为单目没有直接深度，尺度不确定
- 需要先通过两视图几何恢复初始位姿和地图
- 初始化成功后才能稳定进入后续 PnP / BA 流程

### 17.2 VIO 初始化在估什么

一个完整回答通常包括：

- 重力方向
- 速度
- 尺度
- IMU 偏置
- 有时还包括外参

### 17.3 为什么时间同步误差很致命

因为多传感器融合默认假设这些观测属于同一时刻。

一旦不同步，高动态下就会把：

- 视觉约束
- 惯性约束
- 点云几何约束

错误地耦合在一起。

### 17.4 LIO 为什么特别强调去畸变

因为 LiDAR 一帧扫描是分时采集的。

若平台在运动而没有正确 deskew，同一帧点云内部就已经不一致，后面配准和融合都会受影响。

## 18. 一个最小实践建议

如果后面你要自己做一套 VIO / LIO 系统，建议优先保证下面这条顺序：

1. 先确认时间戳可靠  
2. 再做相机内参  
3. 再做相机-IMU 或 LiDAR-IMU 外参  
4. 再检查噪声参数和单位  
5. 最后才去调前端和后端参数

这个顺序通常比“一上来就调优化器参数”有效得多。
