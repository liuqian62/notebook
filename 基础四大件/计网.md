## 计算机网络在学什么

计算机网络关注的是：不同机器之间如何可靠、高效地传输数据。

核心问题包括：

- 怎么寻址
- 怎么分层
- 怎么保证可靠传输
- 怎么控制拥塞
- 应用协议怎么组织请求和响应

## 分层模型

### OSI 七层

- 应用层
- 表示层
- 会话层
- 传输层
- 网络层
- 数据链路层
- 物理层

OSI 更适合作为教学模型。

### TCP/IP 四层

- 应用层
- 传输层
- 网络层
- 网络接口层

工程里更常用 TCP/IP 模型。

### 为什么要分层

- 降低系统复杂度
- 每层职责清晰
- 便于替换和扩展
- 上层屏蔽下层细节

## 应用层

应用层直接面向具体业务协议。

常见协议：

- HTTP / HTTPS
- DNS
- SMTP
- FTP
- WebSocket

### DNS

DNS 的作用是把域名解析成 IP 地址。

常见过程：

- 浏览器先查本地缓存
- 再查系统缓存、hosts
- 然后走递归或迭代查询

## 传输层

传输层负责端到端的数据传输。

核心协议：

- TCP
- UDP

### TCP 和 UDP 的区别

- TCP 面向连接，UDP 无连接
- TCP 可靠传输，UDP 尽力而为
- TCP 有拥塞控制和流量控制，UDP 没有
- TCP 头部更大，UDP 更轻量

### TCP 适用场景

- 对可靠性要求高
- 文件传输
- Web 请求
- 数据不能丢失的场景

### UDP 适用场景

- 更关注实时性
- 音视频
- 直播
- DNS

## 网络层

网络层主要负责主机到主机之间的路由和寻址。

核心协议：

- IP
- ICMP
- ARP

### IP

IP 负责把数据包从源主机送到目标主机，但本身不保证可靠到达。

### ARP

ARP 用于把 IP 地址解析成 MAC 地址，常发生在局域网通信中。

### ICMP

ICMP 常用于网络控制和诊断，例如：

- `ping`
- `traceroute`

## 数据链路层与物理层

### 数据链路层

主要负责：

- 成帧
- 差错检测
- MAC 寻址
- 局域网内传输

### 物理层

物理层关心比特流如何在物理介质上传输，例如电信号、光信号等。

## TCP 核心机制

### 可靠传输

TCP 的可靠性主要依赖：

- 序列号
- 确认应答 ACK
- 超时重传
- 校验和
- 去重和按序重组

### 流量控制

流量控制解决的是“发送方别把接收方压垮”，核心依赖接收窗口。

### 拥塞控制

拥塞控制解决的是“别把网络压垮”。

典型机制：

- 慢启动
- 拥塞避免
- 快重传
- 快恢复

## 三次握手与四次挥手

### 为什么要三次握手

三次握手用于建立连接并确认双方的收发能力都正常。

简化理解：

- 第一次：客户端告诉服务端“我想连你”
- 第二次：服务端告诉客户端“我收到你了，我也能发”
- 第三次：客户端告诉服务端“我也收到你了”

### 为什么不是两次

两次无法充分确认双方都具备收发能力，也不能很好处理历史连接请求报文问题。

### 为什么挥手是四次

TCP 是全双工的，双方关闭连接通常要分别发送 `FIN` 和确认 `ACK`，因此一般需要四次。

### TIME_WAIT

主动关闭连接的一方通常会进入 `TIME_WAIT`。

主要原因：

- 保证最后的 ACK 能被对方收到
- 让旧连接中的报文在网络中自然消失

## HTTP 与 HTTPS

### HTTP

HTTP 是应用层协议，特点：

- 无状态
- 基于请求响应模型
- 常运行在 TCP 之上

### HTTP 常见方法

- `GET`
- `POST`
- `PUT`
- `DELETE`

### HTTP 状态码

- `2xx` 成功
- `3xx` 重定向
- `4xx` 客户端错误
- `5xx` 服务端错误

面试高频：

- `200 OK`
- `301/302`
- `403 Forbidden`
- `404 Not Found`
- `500 Internal Server Error`

### HTTP/1.1、HTTP/2、HTTP/3

- HTTP/1.1 支持长连接，但仍存在队头阻塞问题
- HTTP/2 支持多路复用、头部压缩、服务器推送
- HTTP/3 基于 QUIC，底层改用 UDP，重点优化传输层体验

### HTTPS

HTTPS = HTTP + TLS。

它解决的主要问题：

- 防窃听
- 防篡改
- 防冒充

### HTTPS 握手大意

- 客户端发起握手请求
- 服务端返回证书和协商参数
- 客户端验证证书
- 双方协商对称密钥
- 后续使用对称加密传输数据

## 常见网络问题

### 从输入 URL 到页面显示发生了什么

典型流程：

- URL 解析
- DNS 解析
- 建立 TCP 或 TLS 连接
- 浏览器发送 HTTP 请求
- 服务端处理并返回响应
- 浏览器解析 HTML、CSS、JS
- 页面渲染

### 为什么会出现粘包拆包

因为 TCP 是面向字节流的，没有天然消息边界，应用层必须自己定义协议边界。

### 为什么有了 IP 还需要 MAC

- IP 解决跨网络寻址
- MAC 解决局域网内下一跳寻址

## 常见面试题速答

### TCP 和 UDP 怎么选

- 要可靠性选 TCP
- 要实时性、可接受少量丢包时可选 UDP

### 为什么 TCP 可靠

因为它有序列号、确认应答、重传机制、流量控制和拥塞控制。

### HTTPS 比 HTTP 安全在哪里

HTTPS 在传输层之上加了 TLS，提供加密、完整性校验和身份认证。

### 为什么会有 TIME_WAIT

为了确保连接可靠关闭，并避免旧报文影响后续同名连接。

### 浏览器输入 URL 后为什么先 DNS

因为只有先把域名解析成 IP，客户端才能知道该把请求发到哪台主机。
